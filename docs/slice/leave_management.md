# LLD Specification: Leave Management System

This document outlines the detailed system design, API contracts, domain model, database tables, and concurrency controls for a flexible Leave Management System, optimized for the Slice SDE 2 System Design + LLD round.

---

### ▶ PHASE 1 — OPENING (5 min of talk time)

#### A. CLARIFYING QUESTIONS
1. **Approval Pipeline:** Is leave approval a single manager sign-off, or does it support multi-level workflow pipelines (e.g. Manager -> HR)?
2. **Accrual Schedule:** How do employee leave balances accrue (upfront yearly, monthly, or dynamically calculated)?
3. **Carry Forward:** Do we support carrying forward unused leave balances to the next calendar year, and is there a cap on it?
4. **Validation Constraints:** Can an employee submit overlapping leave requests, and are leaves capped strictly by available balance?

#### B. ASSUMPTIONS & METRICS
* **Workflows:** Multi-level approval pipelines (Manager -> HR).
* **Accrual:** Leaves are accrued monthly (e.g. 1.5 days accrued per month).
* **Units:** Leave duration is tracked in half-day units as integers to prevent float issues.
* **Checks:** Overlapping date ranges are blocked. Balance checks prevent negative balances.

### ▶ PHASE 2 — FULL DESIGN

#### 🏗️ HLD & SERVICE FLOWS

##### A. Core Microservice Boundaries
* `EmployeeService` (DB: `employee_db`) → Employee database, reporting line structure.
* `LeaveService` (DB: `leave_db`) → Leaves application logic, workflow state tracks.
* `AccrualService` (DB: `leave_db`) → Offline worker running balance updates.

##### B. Step-by-Step Service Communication Flow
1. `Employee` —[HTTPS POST /v1/leaves/request]→ `API Gateway`: Submits request with dates, type, and reason.
2. `API Gateway` —[gRPC: checkOverlap]→ `LeaveService`: Validates that dates do not conflict with existing approved requests.
3. `API Gateway` —[gRPC: checkBalance]→ `LeaveService`: Queries balance validity.
4. `LeaveService` —[SQL SELECT]→ `PostgreSQL DB`: Retrieves current employee balances.
5. `LeaveService` —[SQL UPDATE]→ `PostgreSQL DB`: Deducts balance and increments row version (optimistic lock).
6. `LeaveService` —[SQL INSERT]→ `PostgreSQL DB`: Writes `leave_requests` and triggers `leave_approvals` sequence entries.
7. `LeaveService` —[Kafka Event: LEAVE_SUBMITTED]→ `Notification Service`: Asynchronously flags manager queue.

##### C. Infrastructure Topology
* **Load Balancer:** L7 Round-robin balancer.
* **Rate Limiter:** Token bucket limiting request creations (limit: 5 requests/min per employee).
* **Blob Storage:** Medical certificates or attachments stored in S3.
* **Cache Layers:** Redis caching current user balance figures.

#### 🗄️ LLD PART 1 — DATABASE DESIGN & ACCESS PATTERNS

##### A. Detailed Schema Design (PostgreSQL)

```sql
CREATE TABLE leave_types (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    name VARCHAR(50) UNIQUE NOT NULL,
    annual_limit INT NOT NULL,
    carry_forward_limit INT NOT NULL DEFAULT 0,
    is_accrued BOOLEAN NOT NULL DEFAULT TRUE
);

CREATE TABLE leave_balances (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    employee_id BIGINT NOT NULL,
    leave_type_id BIGINT NOT NULL,
    balance INT NOT NULL DEFAULT 0, -- Tracked in half-days (e.g. 1.5 days stored as 3 units)
    version INT NOT NULL DEFAULT 1,
    UNIQUE (employee_id, leave_type_id),
    FOREIGN KEY (leave_type_id) REFERENCES leave_types(id)
);

CREATE TABLE leave_requests (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    employee_id BIGINT NOT NULL,
    leave_type_id BIGINT NOT NULL,
    start_date DATE NOT NULL,
    end_date DATE NOT NULL,
    total_half_days INT NOT NULL CONSTRAINT chk_days_positive CHECK (total_half_days > 0),
    status VARCHAR(25) NOT NULL CONSTRAINT chk_status CHECK (status IN ('SUBMITTED', 'PENDING_APPROVAL', 'APPROVED', 'REJECTED', 'CANCELLED')),
    description VARCHAR(255),
    version INT NOT NULL DEFAULT 1,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    FOREIGN KEY (leave_type_id) REFERENCES leave_types(id)
);

CREATE TABLE leave_approvals (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    request_id BIGINT NOT NULL,
    approver_id BIGINT NOT NULL,
    approval_level INT NOT NULL,
    status VARCHAR(25) NOT NULL CONSTRAINT chk_status CHECK (status IN ('PENDING', 'APPROVED', 'REJECTED')),
    comments TEXT,
    updated_at TIMESTAMP DEFAULT NOW(),
    FOREIGN KEY (request_id) REFERENCES leave_requests(id)
);
```

##### B. Index Design
```
INDEX: idx_requests_dates ON leave_requests(employee_id, start_date, end_date)
Optimized Query: Check overlap: SELECT COUNT(*) FROM leave_requests WHERE employee_id = ? AND (start_date <= :end AND end_date >= :start) AND status NOT IN ('REJECTED', 'CANCELLED')
Without index: Full table scan checking for date overlaps on every submission.
Type: Composite B-tree.
```

##### C. Concurrency & Locking Strategy
Optimistic locking (`version` column) on `leave_balances` prevents concurrent requests from double-deducting balances.
```sql
UPDATE leave_balances
SET balance = balance - :deduct_days,
    version = version + 1
WHERE employee_id = :employee_id
  AND leave_type_id = :leave_type_id
  AND version = :expected_version
  AND balance >= :deduct_days;
```

#### 🏛️ LLD PART 2 — OOD CLASS DESIGN & DESIGN PATTERNS

##### A. Core Class Specifications
```
LeaveRequest (Entity)
  Attributes: id: Long, employeeId: Long, startDate: LocalDate, endDate: LocalDate, totalHalfDays: Integer, status: LeaveStatus

LeaveBalance (Entity)
  Attributes: employeeId: Long, leaveTypeId: Long, balance: Integer, version: Integer

ApprovalHandler (Interface)
  Methods: processApproval(request: LeaveRequest, approverId: long): void
  Implemented by: ManagerApprovalHandler, HRApprovalHandler (Chain of Responsibility Pattern)
```

##### B. Entity Relationships
* **Realization:** `ManagerApprovalHandler` realizes `ApprovalHandler` interface.
* **Association:** `LeaveRequest` maps **One-to-Many** to `LeaveApproval` steps.
* **Composition:** `LeaveRequest` lifecycle is updated using the `LeaveStateMachine`.

##### C. Applied Design Patterns
* **Chain of Responsibility Pattern:** Used for dynamic multi-level approvals. Each approver class processes their level and passes it to the next step.
* **State Pattern:** Governs state validations and blocks unauthorized transitions (e.g. `REJECTED` to `APPROVED`).

##### D. Strict Coding Rules
* Duration and balances are stored as **integers representing half-days** (avoiding floating-point math).

---

#### ⚡ LLD PART 3 — EDGE CASES & SENIOR SIGNALS

1. **Overlapping Date Detection:** Overlaps are detected using the condition `(start_date <= :end_date AND end_date >= :start_date)`. This validates any date collision.
2. **Reverse Deductions:** If a request is rejected at any level in the approval chain, a database transaction is triggered to credit the deducted balance back to `leave_balances`.
3. **Carry Forward Engine:** A cron job runs on Dec 31st. It calculates carry-forwards using `MIN(current_balance, carry_forward_limit)` and resets employee balances.

#### 💬 REVISION & CHALLENGE CARDS

* **Why store leave duration in half-days as integers?**
  > *"Floating-point numbers (0.5, 1.5) are susceptible to binary rounding errors (e.g. 0.5 + 0.5 = 1.0000000000000002). Storing values as integers representing half-day units (e.g. 1 day = 2 units, 0.5 days = 1 unit) guarantees arithmetic precision. We format values at the UI layer."*

* **Why use the Chain of Responsibility Pattern for approvals?**
  > *"Approval steps vary by business (e.g. Manager only, or Manager -> Director -> HR). Hardcoding these rules results in complex nested conditionals. The Chain of Responsibility pattern structures workflows as sequential handlers, making the design clean and extensible."*
