# LLD Specification: Delivery Tracking System

This document outlines the detailed low-level design, API contracts, database schema, high-write location ingestion caching, and concurrency strategy for a real-time Delivery Tracking System, optimized for the Slice SDE 2 System Design + LLD round.

---

### ▶ PHASE 1 — OPENING (5 min of talk time)

#### A. CLARIFYING QUESTIONS
1. **GPS Ingestion Volume:** How frequently do agent mobile devices push coordinates, and what is the target active agent count?
2. **Access Patterns:** Are we storing historical path routes, or is the customer tracking page only concerned with the latest position?
3. **Out-of-order Webhooks:** Can driver transition updates arrive out of sequence, and how do we resolve state machine integrity?
4. **Agent Matching:** Is agent assignment done synchronously inside the order checkout or asynchronously via matching queues?

#### B. ASSUMPTIONS & METRICS
* **Scale:** 10,000 active agents, pushing GPS coordinates every 10 seconds (1,000 writes/sec ingestion load).
* **Storage:** Latest location cached in Redis; final route polyline compressed and archived upon completion.
* **Transition States:** strict order: `ASSIGNED` -> `ACCEPTED` -> `ARRIVED_AT_STORE` -> `PICKED_UP` -> `DELIVERED`/`FAILED`.
* **Accuracy:** Customers query coordinates every 5-10 seconds during tracking.

### ▶ PHASE 2 — FULL DESIGN

#### 🏗️ HLD & SERVICE FLOWS

##### A. Core Microservice Boundaries
* `DeliveryService` (DB: `delivery_db`) → Manages core order state transitions and metadata.
* `LocationService` (DB: `Redis`) → Ingests live location updates and performs geo-searches.
* `DispatchService` (DB: `delivery_db`) → Dispatches matching orders to nearby active drivers.

##### B. Step-by-Step Service Communication Flow
1. `Agent App` —[HTTPS POST /v1/deliveries/locations]→ `API Gateway`: Pushes GPS coordinate payload.
2. `API Gateway` —[gRPC: saveLocation]→ `LocationService`: Forwards coordinates.
3. `LocationService` —[Redis HSET]→ `Redis In-Memory`: Updates the agent's current coordinates hash.
4. `LocationService` —[Redis GEOADD]→ `Redis GeoIndex`: Registers coordinates under active agent geospatial index.
5. `LocationService` —[Kafka Event: PATH_LOGGED]→ `Analytics Service`: Asynchronously logs position logs for path history.
6. `Agent App` —[HTTPS PATCH /v1/deliveries/{id}/status]→ `API Gateway`: Submits transition command.
7. `API Gateway` —[gRPC: updateStatus]→ `DeliveryService`: Processes state alteration.
8. `DeliveryService` —[SQL SELECT FOR UPDATE]→ `PostgreSQL DB`: Locks order row, validates state rules, writes state update, and appends to logs.

##### C. Infrastructure Topology
* **Load Balancer:** L7 Round-robin balancer routing HTTP/WebSocket traffic.
* **Rate Limiter:** Token bucket at Gateway limiting GPS update queries (1 request per 8 seconds per driver).
* **Blob Storage:** Completed route paths compressed to a polyline string and archived in S3.
* **Cache Layers:** Redis is the primary store for live agent coordinate caching.

#### 🗄️ LLD PART 1 — DATABASE DESIGN & ACCESS PATTERNS

##### A. Detailed Schema Design (PostgreSQL)

```sql
CREATE TABLE deliveries (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    order_id BIGINT UNIQUE NOT NULL,
    agent_id BIGINT NOT NULL,
    status VARCHAR(25) NOT NULL CONSTRAINT chk_status CHECK (status IN ('ASSIGNED', 'ACCEPTED', 'PICKED_UP', 'DELIVERED', 'FAILED')),
    pickup_time TIMESTAMP,
    delivery_time TIMESTAMP,
    idempotency_key VARCHAR(64) UNIQUE NOT NULL,
    version INT NOT NULL DEFAULT 1,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE delivery_status_log (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    delivery_id BIGINT NOT NULL,
    status VARCHAR(25) NOT NULL,
    latitude DECIMAL(9,6) NOT NULL,
    longitude DECIMAL(9,6) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    FOREIGN KEY (delivery_id) REFERENCES deliveries(id)
);
```

##### B. NoSQL Decision: Redis Live Location Cache
PostgreSQL is bypassed for raw GPS location pings to protect the database transaction logs from saturation.
```
Database Choice: Redis (In-Memory Hash & GeoIndex)
Key Layout: agent_loc:{agent_id}
Fields: { "lat": 12.9715, "lng": 77.5945, "bearing": 180.5, "updated_at": 178229930 }
TTL: 60 seconds (expires offline drivers)

Geospatial Indexing:
  Command: GEOADD active_agents 77.594562 12.971598 {agent_id}
  Range Query: GEORADIUS active_agents 77.594000 12.971000 5 km WITHDIST ASC LIMIT 10
```

##### C. Concurrency & Locking Strategy
Redis distributed locks are used for status change transactions.
```sql
-- Dynamic status update constraint block (PostgreSQL)
UPDATE deliveries
SET status = :to_status, 
    version = version + 1, 
    updated_at = NOW()
WHERE id = :id 
  AND version = :expected_version
  AND status NOT IN ('DELIVERED', 'FAILED');
```

##### A. Core Class Specifications

##### B. Entity Relationships
* **Realization:** `NearestAgentStrategy` implements `AgentAssignmentStrategy` interface.
* **Association:** `Delivery` maps **One-to-Many** to `DeliveryStatusLog` records.
* **Composition:** `DeliveryStateMachine` is associated with `DeliveryService` state validation.

##### C. Applied Design Patterns
* **Strategy Pattern:** Used for flexible driver matching algorithms (e.g. nearest vs. load balanced).
* **State Pattern:** Governs state validations, blocking illegal transitions (e.g. `ASSIGNED` directly to `DELIVERED`).

#### ⚡ LLD PART 3 — EDGE CASES & SENIOR SIGNALS

1. **Dead Reckoning:** If a client requests driver location but the Redis cache has expired due to connectivity loss, we estimate the current position using the agent's last recorded speed, bearing, and time delta.
2. **OutOfOrder Status Updates:** If updates arrive out of sequence, the `DeliveryStateMachine` rejects the regression (e.g. `PICKED_UP` arriving after a `DELIVERED` status has already committed).
3. **Route Polyline Archival:** During delivery, coordinates are pushed to a Redis list (`route:history:{id}`). Upon transition to `DELIVERED`, the list is retrieved, compressed into an encoded polyline string, saved to S3, and deleted from Redis.

#### 💬 REVISION & CHALLENGE CARDS

* **Why not write live GPS updates to PostgreSQL?**
  > *"Writing 1,000 coordinates/sec to a relational DB results in massive write amplification (WAL updates, page splits, index updates). This degrades performance for critical transactional queries. In-memory Redis writes easily handle 100K requests/sec with low latency."*

* **How do we handle high read loads for active orders?**
  > *"Client tracking pages query coordinates from the Redis cache (`agent_loc:{agent_id}`) directly instead of querying the relational database. This isolates read spikes from the primary database, keeping it fast."*
