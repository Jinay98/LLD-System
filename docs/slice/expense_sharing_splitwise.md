# LLD Specification: Expense Sharing System (Splitwise-style)

This document outlines the detailed low-level design, API contracts, domain class schemas, database tables, and concurrency strategies for a Splitwise-like Expense Sharing System, optimized for the Slice SDE 2 System Design + LLD round.

---

### ▶ PHASE 1 — OPENING (5 min of talk time)

#### A. CLARIFYING QUESTIONS
1. **Split Types:** Do we support multiple split methods: Equal splits, Exact cash amounts, and Percentages?
2. **Settlements:** Is the debt settlement automated (payment integration) or recorded manually by users?
3. **Friendships/Groups:** Must transactions belong to a Group, or can users split expenses directly with another user (non-group)?
4. **Simplification:** Do we calculate minimized transaction paths (Simplify Debts) dynamically on the fly or synchronously during writes?

#### B. ASSUMPTIONS & METRICS
* **Split Types:** Equal, Exact, and Percentage splitting.
* **Simplification:** "Simplify Debts" is an $O(N \log N)$ algorithm evaluated on-demand when users fetch group totals.
* **Currency:** INR stored in paise as integers to avoid rounding float errors.
* **Friendship:** Direct 1-on-1 expenses are supported.

### ▶ PHASE 2 — FULL DESIGN

#### 🏗️ HLD & SERVICE FLOWS

##### A. Core Microservice Boundaries
* `GroupService` (DB: `group_db`) → Manages group creations and user list validation.
* `ExpenseService` (DB: `expense_db`) → Registers and stores bills, validate split models.
* `BalanceService` (DB: `balance_db`) → Maintains running debt balances and executes simplification.

##### B. Step-by-Step Service Communication Flow
1. `Client` —[HTTPS POST /v1/expenses]→ `API Gateway`: Submits expense details, split mappings, and idempotency key.
2. `API Gateway` —[gRPC: validateGroup]→ `GroupService`: Verifies if the group exists and participants are members.
3. `API Gateway` —[gRPC: addExpense]→ `ExpenseService`: Forwards the request parameters.
4. `ExpenseService` —[Strategy validation]→ `ExpenseService`: Validates split inputs against rules (e.g. sum checks).
5. `ExpenseService` —[SQL Transaction]→ `PostgreSQL DB`: Inserts expense record and split rows.
6. `ExpenseService` —[Kafka Event: EXPENSE_CREATED]→ `BalanceService`: Asynchronously emits creation event.
7. `BalanceService` —[SQL SELECT FOR UPDATE]→ `PostgreSQL DB`: Acquires exclusive locks on pairwise relations in strict user order.
8. `BalanceService` —[SQL UPDATE]→ `PostgreSQL DB`: Updates cached balance cells representing who owes whom.

##### C. Infrastructure Topology
* **Load Balancer:** L7 LB routing request traffic to application nodes.
* **Rate Limiter:** Token bucket limiting card checkout or entry creation spikes (15 requests/min per user).
* **Blob Storage:** Receipt invoice images uploaded and stored in S3, linked via URL inside metadata.
* **Cache Layers:** Redis caching user balance structures, invalidated when new expenses are compiled.

#### 🗄️ LLD PART 1 — DATABASE DESIGN & ACCESS PATTERNS

##### A. Detailed Schema Design (PostgreSQL)

```sql
CREATE TABLE groups (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    name VARCHAR(100) NOT NULL,
    created_by BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE group_members (
    group_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    joined_at TIMESTAMP DEFAULT NOW(),
    PRIMARY KEY (group_id, user_id),
    FOREIGN KEY (group_id) REFERENCES groups(id)
);

CREATE TABLE expenses (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    group_id BIGINT, -- NULL for non-group expenses
    paid_by BIGINT NOT NULL,
    total_amount BIGINT NOT NULL CONSTRAINT chk_amount_positive CHECK (total_amount > 0),
    split_type VARCHAR(20) NOT NULL CONSTRAINT chk_split_type CHECK (split_type IN ('EQUAL', 'EXACT', 'PERCENTAGE')),
    description VARCHAR(255),
    idempotency_key VARCHAR(64) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    FOREIGN KEY (group_id) REFERENCES groups(id)
);

CREATE TABLE expense_splits (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    expense_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    amount_owed BIGINT NOT NULL CONSTRAINT chk_owed_positive CHECK (amount_owed >= 0),
    percentage DECIMAL(5,2),
    FOREIGN KEY (expense_id) REFERENCES expenses(id)
);

CREATE TABLE user_balances (
    user_a BIGINT NOT NULL,
    user_b BIGINT NOT NULL,
    balance BIGINT NOT NULL DEFAULT 0,
    version INT NOT NULL DEFAULT 1,
    PRIMARY KEY (user_a, user_b),
    CONSTRAINT chk_user_order CHECK (user_a < user_b)
);
```

##### B. Index Design
```
INDEX: idx_expense_group ON expenses(group_id, created_at DESC)
Optimized Query: SELECT * FROM expenses WHERE group_id = ? ORDER BY created_at DESC
Without index: Full table scan on expenses every time a user opens a group history feed.
Type: Composite B-tree.
```

##### C. Concurrency & Locking Strategy
Pessimistic row locking is applied to pairwise relationships in `user_balances`. Transactions lock rows in ascending order based on ID keys to prevent deadlocks during simultaneous entries.
```sql
-- Lock pair rows dynamically sorted
SELECT balance FROM user_balances 
WHERE user_a = LEAST(:user1, :user2) 
  AND user_b = GREATEST(:user1, :user2) 
FOR UPDATE;

-- Update the record balance cell
UPDATE user_balances 
SET balance = balance + :delta_amount
WHERE user_a = :user_a AND user_b = :user_b;
```

#### 🏛️ LLD PART 2 — OOD CLASS DESIGN & DESIGN PATTERNS

##### A. Core Class Specifications
```
Expense (Entity)
  Attributes: id: Long, groupId: Long, paidBy: Long, totalAmount: BigDecimal, splitType: SplitType
  Methods: validate()

Split (Class)
  Attributes: userId: Long, shareAmount: BigDecimal

SplitStrategy (Interface)
  Methods: validateSplits(amount: BigDecimal, splits: List<Split>): boolean
  Implemented by: EqualSplitStrategy, ExactSplitStrategy, PercentageSplitStrategy

BalanceSheetManager (Service)
  Attributes: balanceRepo: BalanceRepository
  Methods: simplifyDebts(groupId: long): List<DebtSettlement>
```

##### B. Entity Relationships
* **Realization:** `EqualSplitStrategy` realizes `SplitStrategy`.
* **Composition:** `Expense` owns instances of `Split` objects. They cannot exist without the parent.
* **Association:** `Expense` maps **One-to-Many** to `Split` entries.

##### C. Applied Design Patterns
* **Strategy Pattern:** `SplitStrategy` encapsulates parsing rules per splitting type.
* **Factory Pattern:** `SplitStrategyFactory` generates matching strategy logic classes.

##### D. Strict Coding Rules
* Precision balance math uses `BigDecimal`.
* Rounded split remainders are distributed back to the payer (see Part 3).

#### ⚡ LLD PART 3 — EDGE CASES & SENIOR SIGNALS

1. **Division Remainders:** Dividing ₹10.00 among 3 users equally results in 333.33 paise each. We calculate the remainder (1 paisa) using `total_amount % users_count` and allocate it directly to the payer's share to maintain ledger balance equations.
2. **Debt Simplification Heap Engine:** We construct net positions (credit/debit) for users. Creditors are pushed to a Max-Heap, and debtors are pushed to a Min-Heap. Popping matching heads and offsetting balances reduces $N$ user pathways to a maximum of $N-1$ transaction settlements.
3. **Soft Delete Reversals:** If a user cancels or soft-deletes a recorded bill, we invoke a transaction block that locks `user_balances` and reverses the calculation, restoring previous balances.

#### 💬 REVISION & CHALLENGE CARDS

* **Why maintain a pairwise balance table instead of calculating dynamically?**
  > *"Calculating balances dynamically requires running `SUM` joins over the `expense_splits` and `expenses` tables. As order history scales, this query becomes extremely slow. Pairwise records provide $O(1)$ read performance. We handle concurrent write safety via row-level locks."*

* **How do we handle multi-currency transactions?**
  > *"We choose a base currency (INR) for user relationships. If an expense is entered in USD, we query the conversion rate api at checkout, convert values to base INR paise, write calculations to database entries, and log original input fields in a metadata payload."*
