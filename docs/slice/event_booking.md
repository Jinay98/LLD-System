# LLD Specification: Event Booking System (BookMyShow-style)

This document outlines the detailed low-level design, API contracts, domain model, database tables, and concurrency controls for a high-traffic Event & Ticket Booking System, optimized for the Slice SDE 2 System Design + LLD round.

---

### ▶ PHASE 1 — OPENING (5 min of talk time)

#### A. CLARIFYING QUESTIONS
1. **Seat Selection Model:** Do we support choosing specific seat coordinates (e.g. Row H, Seat 15), or only general admission counts?
2. **Hold TTL:** How long are seats locked for a user during checkout before they are automatically released back to the pool?
3. **Flash Sales:** Do we expect flash sales (e.g. concert tickets) where thousands of users compete for a small pool of seats?
4. **Group Bookings:** Can a user book multiple seats in a single checkout operation, and what happens if only some of those seats are available?

#### B. ASSUMPTIONS & METRICS
* **Hold TTL:** Seats are locked for 5 minutes during checkout.
* **Scale:** Highly concurrent flash checkout (e.g., 50K users for 10K seats, peaking at 1,000 requests/sec).
* **Seating:** Specific seat-based selection grid.
* **Integrations:** Payment callbacks trigger booking confirmation asynchronously.

### ▶ PHASE 2 — FULL DESIGN

#### 🏗️ HLD & SERVICE FLOWS

##### A. Core Microservice Boundaries
* `CatalogService` (DB: `catalog_db`) → Read-heavy catalog displaying shows, movies, and pricing.
* `BookingService` (DB: `booking_db`) → Handles seat reservations, checkout states, and ticket locking.
* `PaymentService` (DB: `payment_db`) → Manages payments and webhooks.

##### B. Step-by-Step Service Communication Flow
1. `Client` —[HTTPS POST /v1/bookings/reserve]→ `API Gateway`: Submits show ID and seat list.
2. `API Gateway` —[gRPC: holdSeats]→ `BookingService`: Starts hold flow.
3. `BookingService` —[Redis SETNX Loop]→ `Redis Hold Cache`: Evaluates and locks each seat with a 5-minute TTL.
4. `BookingService` —[SQL INSERT]→ `PostgreSQL DB`: Inserts a `PENDING` booking row.
5. `Client` —[HTTPS POST /v1/payments/charge]→ `PaymentService`: Executes payment.
6. `PaymentService` —[HTTPS webhook callback]→ `BookingService`: Notifies of payment success.
7. `BookingService` —[SQL Transaction]→ `PostgreSQL DB`: Updates `show_seats` to `BOOKED` and sets booking status to `CONFIRMED`.
8. `BookingService` —[Kafka Event: TICKET_CONFIRMED]→ `Notification Service`: Asynchronously triggers PDF generation and sends tickets to the user.

##### C. Infrastructure Topology
* **Load Balancer:** L7 LB routing catalog reads to replicas and booking writes to active writers.
* **Rate Limiter:** Token bucket at API gateway limiting checkout queries (limit: 5 checkouts/min per user).
* **Blob Storage:** Generated ticket PDF/QR images stored in S3.
* **Cache Layers:** Redis is used for fast seat-hold locks to bypass database lock contention.

#### 🗄️ LLD PART 1 — DATABASE DESIGN & ACCESS PATTERNS

##### A. Detailed Schema Design (PostgreSQL)

```sql
CREATE TABLE shows (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    movie_id BIGINT NOT NULL,
    screen_id BIGINT NOT NULL,
    start_time TIMESTAMP NOT NULL,
    end_time TIMESTAMP NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE show_seats (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    show_id BIGINT NOT NULL,
    seat_number VARCHAR(10) NOT NULL,
    seat_type VARCHAR(20) NOT NULL CONSTRAINT chk_seat_type CHECK (seat_type IN ('NORMAL', 'PREMIUM', 'VIP')),
    price BIGINT NOT NULL CONSTRAINT chk_price_positive CHECK (price > 0),
    status VARCHAR(20) NOT NULL CONSTRAINT chk_status CHECK (status IN ('AVAILABLE', 'LOCKED', 'BOOKED')),
    locked_by BIGINT,
    locked_until TIMESTAMP,
    version INT NOT NULL DEFAULT 1,
    UNIQUE (show_id, seat_number),
    FOREIGN KEY (show_id) REFERENCES shows(id)
);

CREATE TABLE bookings (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    user_id BIGINT NOT NULL,
    show_id BIGINT NOT NULL,
    status VARCHAR(20) NOT NULL CONSTRAINT chk_status CHECK (status IN ('PENDING', 'CONFIRMED', 'CANCELLED')),
    total_price BIGINT NOT NULL CONSTRAINT chk_price_positive CHECK (total_price > 0),
    created_at TIMESTAMP DEFAULT NOW(),
    FOREIGN KEY (show_id) REFERENCES shows(id)
);
```

##### B. NoSQL Decision: Redis Seat Lock Hold (Flash Sale Guard)
In flash sales, DB row-level locking fails due to thread pool starvation. We offload holds to Redis:
```
Database Choice: Redis
Key Schema: seat_hold:{show_id}:{seat_number}
Value: user_id
TTL: 300 seconds (5 minutes)

Hold Command: SET seat_hold:{show_id}:{seat_number} {user_id} NX EX 300
  NX = Set only if key does not exist. (Atomic lock check-and-set).
  Returns: "OK" on success, null on lock failure.
```

##### C. Concurrency & Locking Strategy
Once Redis hold is validated, the database status confirmation is committed in a transaction using optimistic updates:
```sql
UPDATE show_seats
SET status = 'BOOKED', 
    locked_by = NULL, 
    locked_until = NULL, 
    version = version + 1
WHERE show_id = :show_id 
  AND seat_number IN (:seats)
  AND (status = 'AVAILABLE' OR (status = 'LOCKED' AND locked_by = :user_id));
```

#### ⚡ LLD PART 3 — EDGE CASES & SENIOR SIGNALS

1. **Partial Hold Rollback:** If a user reserves H-1, H-2, and H-3, and Redis locks H-1 and H-2 but fails on H-3, we catch the exception and immediately delete H-1 and H-2 Redis keys to prevent orphaned locks.
2. **Expired Payments Refund:** If a payment success callback arrives after the 5-minute hold expires, and another user has booked the seats, the system marks the transaction `REJECTED` and triggers an automated refund process.
3. **Database Lock Cleanup Daemon:** A cron job runs every minute to scan the DB and reset any expired locks (`status = 'LOCKED' AND locked_until < NOW()`) back to `AVAILABLE`.

#### 💬 REVISION & CHALLENGE CARDS

* **Why use Redis for seat holds instead of DB transaction locks?**
  > *"Flash sale traffic saturates PostgreSQL connection pools quickly because locks are held while awaiting payment responses. Redis handles holds in-memory at 100K+ requests/sec, protecting the database from load spikes."*

* **How do we handle double booking if Redis falls over?**
  > *"We configure Redis with Master-Replica replication and Sentinel for auto-failover. As a secondary guard, the database `show_seats` table enforces a unique constraint index on `(show_id, seat_number)` and uses optimistic version checks."*
