# LLD Specification: Payment Aggregator & UPI Gateway

This document outlines the detailed system design, API contracts, sequence flows, database schema, and concurrency strategy for integrating **Payment Aggregators and UPI Gateways**, optimized for the Slice SDE 2 System Design + LLD round.

---

### ▶ PHASE 1 — OPENING (5 min of talk time)

#### A. CLARIFYING QUESTIONS
1. **Aggregator Strategy:** Are we integrating directly with banks via UPI PSP specs, or routing through aggregators (e.g. Razorpay, Cashfree)?
2. **Callback Handling:** Do we get real-time webhook updates, or do we poll the aggregator API for payment outcomes?
3. **Failover Routing:** If Aggregator A fails during checkout, should we dynamically route the client to Aggregator B?
4. **Expiry Window:** How long does a UPI payment collect request remain active before expiring?

#### B. ASSUMPTIONS & METRICS
* **Routing:** Dynamically route checkout traffic between 2 active aggregators.
* **Scale:** Average 100 TPS, Peak 300 TPS.
* **Expiry:** UPI Collect request timeout is set to 10 minutes.
* **Callbacks:** Aggregator webhooks are the primary driver of updates, backed by a poller cron.

### ▶ PHASE 2 — FULL DESIGN

#### 🏗️ HLD & SERVICE FLOWS

##### A. Core Microservice Boundaries
* `CheckoutService` (DB: `payment_db`) → Creates checkouts, maps idempotency, selects dynamic aggregator gateways.
* `CallbackService` (DB: `payment_db`) → Validates webhooks and updates transaction state.
* `ReconcileService` (DB: `payment_db`) → Background worker verifying pending checkouts via gateway queries.

##### B. Step-by-Step Service Communication Flow
1. `Client` —[HTTPS POST /v1/payments/checkout]→ `API Gateway`: Submits purchase details and VPA.
2. `API Gateway` —[gRPC: routeGateway]→ `CheckoutService`: Processes selection rules and saves initial `INITIATED` payment row to DB.
3. `CheckoutService` —[HTTPS POST: createOrder]→ `Aggregator API`: Registers checkout and receives internal platform reference ID.
4. `CheckoutService` —[SQL UPDATE]→ `PostgreSQL DB`: Updates transaction status to `PENDING` with external reference.
5. `Client App` —[UPI Intent / App Redirect]→ `UPI App`: Customer authorizes payment using personal PIN.
6. `Aggregator Webhook` —[HTTPS POST /v1/payments/webhooks]→ `CallbackService`: Callback notifies system of state capture.
7. `CallbackService` —[SQL SELECT FOR UPDATE]→ `PostgreSQL DB`: Checks state limits and changes record status to `SUCCESS`.
8. `CallbackService` —[Kafka Event: PAYMENT_SUCCESS]→ `Order/Ledger Service`: Asynchronously updates order systems.

##### C. Infrastructure Topology
* **Load Balancer:** Layer 7 (Application) LB using IP hashing for sticky customer routing if WebSocket polling is configured.
* **Rate Limiter:** Token bucket limiting checkout requests (limit: 5 checkouts/min per card/account).
* **Blob Storage:** Webhook raw payloads archived to S3 for debugging audits.
* **Cache Layers:** Redis distributed lock tracking webhook dedup checks.

#### 🗄️ LLD PART 1 — DATABASE DESIGN & ACCESS PATTERNS

##### A. Detailed Schema Design (PostgreSQL)

```sql
CREATE TABLE payment_transactions (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    order_id VARCHAR(64) UNIQUE NOT NULL,
    amount BIGINT NOT NULL CONSTRAINT chk_amount_positive CHECK (amount > 0),
    status VARCHAR(20) NOT NULL CONSTRAINT chk_status CHECK (status IN ('INITIATED', 'PENDING', 'SUCCESS', 'FAILED', 'EXPIRED')),
    aggregator VARCHAR(50) NOT NULL,
    external_transaction_id VARCHAR(128) UNIQUE,
    idempotency_key VARCHAR(64) UNIQUE NOT NULL,
    version INT NOT NULL DEFAULT 1,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE payment_events (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    transaction_id BIGINT NOT NULL,
    from_status VARCHAR(20) NOT NULL,
    to_status VARCHAR(20) NOT NULL,
    raw_payload JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    FOREIGN KEY (transaction_id) REFERENCES payment_transactions(id)
);
```

##### B. Index Design
```
INDEX: idx_payments_status_update ON payment_transactions(status, updated_at)
Optimized Query: SELECT * FROM payment_transactions WHERE status = 'PENDING' AND updated_at < ?
Without index: Background reconciliation daemon executes slow full table scans looking for stuck payments.
Type: Composite B-tree.
```

##### C. Concurrency & Locking Strategy
To handle concurrent callback requests and prevent client redirect threads from overwriting state, we lock records using optimistic locking (`version` column) and restrict changes out of terminal states.
```sql
-- Execution during webhook callback ingestion
UPDATE payment_transactions
SET status = :to_status, 
    external_transaction_id = :ext_id, 
    version = version + 1, 
    updated_at = NOW()
WHERE id = :id 
  AND version = :expected_version
  AND status = 'PENDING'; -- Prevents terminal state transitions
```

#### 🏛️ LLD PART 2 — OOD CLASS DESIGN & DESIGN PATTERNS

##### A. Core Class Specifications
```
PaymentTransaction (Entity)
  Attributes: id: Long, orderId: String, amount: BigDecimal, status: PaymentStatus, version: Integer

PaymentGateway (Interface)
  Methods: createPayment(orderId: String, amount: BigDecimal, vpa: String): GatewayResult
           queryStatus(externalId: String): GatewayStatusResult
  Implemented by: RazorpayGateway, CashfreeGateway (Strategy Pattern)

PaymentOrchestrator (Service)
  Attributes: gatewayRegistry: GatewayRegistry, paymentRepo: PaymentRepository
  Methods: checkout(req: CheckoutRequest): CheckoutResult
```

##### B. Entity Relationships
* **Realization:** `RazorpayGateway` and `CashfreeGateway` implement `PaymentGateway` interface.
* **Association:** `PaymentTransaction` maps **One-to-Many** to `PaymentEvent` entries.
* **Composition:** `PaymentOrchestrator` governs the processing workflow of `PaymentTransaction`.

##### C. Applied Design Patterns
* **Strategy Pattern:** Used dynamically for choosing the best available aggregator via `PaymentGateway` implementations.
* **State Pattern:** Governs state shifts: `INITIATED` -> `PENDING` -> `SUCCESS`/`FAILED`/`EXPIRED`. Terminal states reject transitions.

##### D. Strict Coding Rules
* Precision amounts utilizing `BigDecimal`.
* Signature validations matching security codes utilize HMAC calculations.

#### ⚡ LLD PART 3 — EDGE CASES & SENIOR SIGNALS

1. **Webhook Deduping:** We execute a Redis lock using `webhook:txn:{ext_id}` for 5 minutes. If a duplicate callback arrives from the gateway, it fails to acquire the Redis lock and returns early.
2. **Terminal State Integrity:** If a payment expires, and a late `payment.captured` callback arrives, we block DB updates to `SUCCESS`. Instead, we write to a manual reconciliation queue for refunds.
3. **Dynamic Failover Circuit Breaker:** If an aggregator fails 3 consecutive checkout registrations, we mark the provider offline in the cache and route subsequent checkouts to the fallback gateway.

#### 💬 REVISION & CHALLENGE CARDS

* **What happens if webhooks fail entirely?**
  > *"We run a background Reconciliation cron job every 5 minutes. It fetches payments stuck in the `PENDING` state and queries the aggregator status endpoint directly, updating database states based on response payloads."*

* **How do we secure webhook callbacks?**
  > *"Every webhook requires HMAC signature verification using a shared gateway secret. We compute the SHA256 signature from the raw payload body and reject updates if it doesn't match the signature provided in the headers."*
