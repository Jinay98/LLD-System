# LLD Specification: Task Management System

This document outlines the detailed system design, API contracts, domain model, database schemas, and concurrency controls for a resilient Task Management System, optimized for the Slice SDE 2 System Design + LLD round.

---

### ▶ PHASE 1 — OPENING (5 min of talk time)

#### A. CLARIFYING QUESTIONS
1. **Assignee count:** Can a task be assigned to multiple users, or is it strictly 1-to-1?
2. **Sub-tasks:** Do we support sub-tasks and task dependency trees (e.g. Blocked status)?
3. **State Machine Transitions:** Can any state move to any state (e.g. `COMPLETED` back to `IN_PROGRESS`), or is the transition path restricted?
4. **Task Assignment Locking:** Do we need to prevent two managers from assigning the same task to different people at the same time?

#### B. ASSUMPTIONS & METRICS
* **Assignee:** Exactly one assignee per task.
* **Hierarchy:** 1-level parent-child sub-task hierarchy.
* **State Machine:** Restricted flow: `CREATED` -> `ASSIGNED` -> `IN_PROGRESS` -> `COMPLETED`/`CANCELLED`.
* **Scale:** 500K tasks, low contention updates per task profile.

### ▶ PHASE 2 — FULL DESIGN

#### 🏗️ HLD & SERVICE FLOWS

##### A. Core Microservice Boundaries
* `TaskService` (DB: `task_db`) → Handles task creations, modifications, and assignments.
* `WorkflowService` (DB: `task_db`) → Validates status transitions and handles dependencies.

##### B. Step-by-Step Service Communication Flow
1. `Client` —[HTTPS PATCH /v1/tasks/{taskId}/status]→ `API Gateway`: Submits request with target state, version, and reason.
2. `API Gateway` —[gRPC: checkAccess]→ `UserService`: Validates requester roles and permissions.
3. `API Gateway` —[gRPC: updateStatus]→ `TaskService`: Forwards transition query.
4. `TaskService` —[SQL SELECT]→ `PostgreSQL DB`: Retrieves task entity details.
5. `TaskService` —[State Machine Check]→ `TaskService`: Validates transition path using the State Pattern.
6. `TaskService` —[SQL UPDATE]→ `PostgreSQL DB`: Updates state, increments version, and asserts version matches `expected_version`.
7. `TaskService` —[SQL INSERT]→ `PostgreSQL DB`: Adds row to `task_status_history` table in the same transaction block.
8. `TaskService` —[Kafka Event: TASK_UPDATED]→ `Notification Service`: Asynchronously emits state update notification.

##### C. Infrastructure Topology
* **Load Balancer:** L7 Round-robin load balancer.
* **Rate Limiter:** Token bucket limiting bulk task assignment queries (limit: 30 requests/min per user).
* **Blob Storage:** Attached task documents uploaded to S3, linked via document URI.
* **Cache Layers:** Redis cache storing team task dashboard data with short TTLs.

#### 🗄️ LLD PART 1 — DATABASE DESIGN & ACCESS PATTERNS

##### A. Detailed Schema Design (PostgreSQL)

```sql
CREATE TABLE tasks (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    title VARCHAR(255) NOT NULL,
    description TEXT,
    status VARCHAR(20) NOT NULL CONSTRAINT chk_status CHECK (status IN ('CREATED', 'ASSIGNED', 'IN_PROGRESS', 'COMPLETED', 'CANCELLED')),
    priority VARCHAR(20) NOT NULL CONSTRAINT chk_priority CHECK (priority IN ('LOW', 'MEDIUM', 'HIGH', 'CRITICAL')),
    creator_id BIGINT NOT NULL,
    assignee_id BIGINT,
    parent_task_id BIGINT,
    due_date TIMESTAMP,
    version INT NOT NULL DEFAULT 1,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    FOREIGN KEY (parent_task_id) REFERENCES tasks(id)
);

CREATE TABLE task_status_history (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    task_id BIGINT NOT NULL,
    from_status VARCHAR(20) NOT NULL,
    to_status VARCHAR(20) NOT NULL,
    changed_by BIGINT NOT NULL,
    reason TEXT,
    created_at TIMESTAMP DEFAULT NOW(),
    FOREIGN KEY (task_id) REFERENCES tasks(id)
);
```

##### B. Index Design
```
INDEX: idx_tasks_assignee_status ON tasks(assignee_id, status)
Optimized Query: SELECT * FROM tasks WHERE assignee_id = ? AND status = 'IN_PROGRESS'
Without index: Full table scan on tasks every time a user views their dashboard.
Type: Composite B-tree.

INDEX: idx_tasks_due_date_status ON tasks(due_date, status)
Optimized Query: SELECT * FROM tasks WHERE due_date < NOW() AND status NOT IN ('COMPLETED', 'CANCELLED')
Without index: Full scan for background SLA overdue check cron.
Type: Composite B-tree.
```

##### C. Concurrency & Locking Strategy
Optimistic locking (`version` column) is used because concurrent writes on a single task are rare. It prevents transaction pool blockages.
```sql
-- Executed inside service transaction
UPDATE tasks
SET status = :new_status,
    version = version + 1,
    updated_at = NOW()
WHERE id = :task_id 
  AND version = :expected_version;
-- If rows updated == 0 -> throw VersionConflictException -> ROLLBACK
```

#### 🏛️ LLD PART 2 — OOD CLASS DESIGN & DESIGN PATTERNS

##### A. Core Class Specifications
```
Task (Entity)
  Attributes: id: Long, title: String, status: TaskStatus, version: Integer, parentTaskId: Long
  Methods: transitionTo(target: TaskStatus, stateMachine: TaskStateMachine)

TaskStateMachine (Domain)
  Attributes: states: Map<TaskStatus, TaskState>
  Methods: transition(task: Task, target: TaskStatus)

TaskState (Interface)
  Methods: validateTransition(target: TaskStatus)
```

##### B. Entity Relationships
* **Realization:** `InProgressState` implements `TaskState`.
* **Association:** `Task` maps **One-to-Many** to `TaskStatusHistory`.
* **Aggregation:** `Task` holds an optional reference to a parent `Task` (hierarchy mapping).

##### C. Applied Design Patterns
* **State Pattern:** Encapsulates transition logic and guards into dedicated state classes, preventing messy conditional logic in services.
* **Observer Pattern:** Decouples core task updates from async services (SMS, slack hooks) using event buses.

---

#### ⚡ LLD PART 3 — EDGE CASES & SENIOR SIGNALS

1. **Auto-Retry Mechanism:** On catching `OptimisticLockingFailureException`, the application service catches the error, re-reads the entity, re-applies the validation, and retries up to 3 times before returning a `409` conflict to the client.
2. **Cascading Cancel Transitions:** If a parent task moves to `CANCELLED`, we run a database transaction that recursively cancels all open child sub-tasks to prevent orphaned processing.
3. **SLA Overdue Poller:** A background daemon checks `idx_tasks_due_date_status` every 5 minutes and emits alerts to the managers for any overdue tasks.

#### 💬 REVISION & CHALLENGE CARDS

* **Why use Optimistic over Pessimistic locking for tasks?**
  > *"Tasks have low write-contention (usually only the assignee or creator updates them). Optimistic locking is light, doesn't hold DB connection locks, and scales cleanly. Returning a version conflict is acceptable UX for rare collision updates."*

* **Why is the State Pattern preferred over simple enums?**
  > *"When state logic grows (e.g. adding notifications, security validations, or automation triggers on transition), simple enum switch statements expand into unmaintainable spaghetti code. The State Pattern encapsulates this logic inside dedicated state classes, adhering to SRP."*
