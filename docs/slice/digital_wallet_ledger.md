# LLD Specification: Digital Wallet & Ledger System

This document outlines the detailed system design, API contracts, class relationships, and database schema for the **Digital Wallet & Ledger System**, optimized for the Slice SDE 2 System Design + LLD round.

---

### ▶ PHASE 1 — OPENING (5 min of talk time)

#### A. CLARIFYING QUESTIONS
1. **Transaction Direction:** Do we support Peer-to-Peer (wallet-to-wallet) transfers only, or also external top-ups (bank-to-wallet) and payouts (wallet-to-bank)?
2. **Multi-currency support:** Is the wallet multi-currency or single-currency (INR)?
3. **Negative Balance:** Can a wallet balance drop below zero (e.g. for credit overdrafts), or is zero a hard constraint?
4. **Limits Check:** Are daily/monthly spend limits verified synchronously inside the transfer transaction or asynchronously?

#### B. ASSUMPTIONS & METRICS
* **Scale:** 1M Daily Active Users (DAU), Peak transaction throughput of 200 TPS.
* **Consistency:** Strict transactional ACID consistency; wallet balances must match the sum of ledger entries.
* **Currency:** Stored in paise (INR) using integers to eliminate rounding issues.
* **Overdraft:** Balances cannot drop below zero.

### ▶ PHASE 2 — FULL DESIGN

#### 🏗️ HLD & SERVICE FLOWS

##### A. Core Microservice Boundaries
* `UserService` (DB: `user_db`) → User metadata, authentication, KYC checks.
* `WalletService` (DB: `wallet_db`) → Balance check/update operations, append-only double-entry ledger.
* `LimitService` (DB: `limit_db`) → Tracks daily/monthly transfer limits.

##### B. Step-by-Step Service Communication Flow
1. `Client` —[HTTPS POST /v1/wallets/transfers]→ `API Gateway`: Submits payload with JWT and unique Idempotency-Key.
2. `API Gateway` —[gRPC: authenticate]→ `UserService`: Validates token and checks if user profile is active.
3. `API Gateway` —[gRPC: checkLimits]→ `LimitService`: Verifies that the transfer amount does not exceed the sender's daily transaction limit.
4. `API Gateway` —[gRPC: executeTransfer]→ `WalletService`: Forwards the request parameters.
5. `WalletService` —[SQL SELECT FOR UPDATE]→ `PostgreSQL DB`: Acquires exclusive row locks on sender and receiver wallets in sorted order.
6. `WalletService` —[SQL INSERT]→ `PostgreSQL DB`: Adds double-entry ledger rows and updates cached wallet balances.
7. `WalletService` —[Kafka Event: TXN_COMMITTED]→ `Notification Service`: Asynchronously publishes event to send SMS and email confirmations.

##### C. Infrastructure Topology
* **Load Balancer:** Layer 7 (Application) LB using round-robin routing with path-based rules.
* **Rate Limiter:** Redis-based token bucket algorithm sitting at the API Gateway layer (rate limit: 50 requests/min per user profile).
* **Blob Storage:** Monthly statement PDFs archived and stored in AWS S3 with read-only signed URLs.
* **Cache Layers:** Redis cache for fast point lookup of wallet balances with dynamic TTL invalidation upon mutations.

#### 🗄️ LLD PART 1 — DATABASE DESIGN & ACCESS PATTERNS

##### A. Detailed Schema Design (PostgreSQL)

```sql
CREATE TABLE wallets (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    user_id BIGINT UNIQUE NOT NULL,
    balance BIGINT NOT NULL DEFAULT 0 CONSTRAINT chk_balance_positive CHECK (balance >= 0),
    version INT NOT NULL DEFAULT 1,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE ledger_entries (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    wallet_id BIGINT NOT NULL,
    amount BIGINT NOT NULL CONSTRAINT chk_amount_positive CHECK (amount > 0),
    entry_type VARCHAR(10) NOT NULL CONSTRAINT chk_entry_type CHECK (entry_type IN ('DEBIT', 'CREDIT')),
    transaction_ref VARCHAR(64) NOT NULL,
    description VARCHAR(255),
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    FOREIGN KEY (wallet_id) REFERENCES wallets(id)
);

CREATE TABLE idempotency_records (
    idempotency_key VARCHAR(64) PRIMARY KEY,
    status VARCHAR(20) NOT NULL CONSTRAINT chk_status CHECK (status IN ('PROCESSING', 'SUCCESS', 'FAILED')),
    response_payload TEXT,
    created_at TIMESTAMP DEFAULT NOW(),
    expires_at TIMESTAMP NOT NULL
);
```

##### B. Index Design
```
INDEX: idx_ledger_wallet_time ON ledger_entries(wallet_id, created_at DESC)
Optimized Query: SELECT * FROM ledger_entries WHERE wallet_id = ? ORDER BY created_at DESC LIMIT 20
Without index: Full table scan and file sort on every ledger page fetch.
Type: Composite B-tree.

INDEX: idx_ledger_tx_ref ON ledger_entries(transaction_ref)
Optimized Query: SELECT * FROM ledger_entries WHERE transaction_ref = ?
Without index: Full table scan to link credit and debit legs of a transaction.
Type: B-tree.
```

##### C. Concurrency & Locking Strategy
Pessimistic row-level locking (`SELECT ... FOR UPDATE`) is used for wallet balance updates to prevent race conditions during concurrent transfer requests.
```sql
-- Executed inside a single transaction
BEGIN;
  -- Lock rows in ascending ID order to prevent deadlocks
  SELECT balance FROM wallets WHERE id = LEAST(:sender_id, :receiver_id) FOR UPDATE;
  SELECT balance FROM wallets WHERE id = GREATEST(:sender_id, :receiver_id) FOR UPDATE;

  -- Validate sender balance checks in application code...

  UPDATE wallets SET balance = balance - :amount, updated_at = NOW() WHERE id = :sender_id;
  UPDATE wallets SET balance = balance + :amount, updated_at = NOW() WHERE id = :receiver_id;

  INSERT INTO ledger_entries (wallet_id, amount, entry_type, transaction_ref, created_at)
  VALUES (:sender_id, :amount, 'DEBIT', :txn_ref, NOW());
  INSERT INTO ledger_entries (wallet_id, amount, entry_type, transaction_ref, created_at)
  VALUES (:receiver_id, :amount, 'CREDIT', :txn_ref, NOW());
COMMIT;
```

##### A. Core Class Specifications
```
Wallet (Entity)
  Attributes: id: Long, userId: Long, balance: BigDecimal, version: Integer
  Methods: debit(amount: BigDecimal), credit(amount: BigDecimal)

LedgerEntry (Entity)
  Attributes: id: Long, walletId: Long, amount: BigDecimal, entryType: EntryType, txnRef: String

WalletService (Service)
  Attributes: walletRepo: WalletRepository, ledgerRepo: LedgerRepository, idempotencyRepo: IdempotencyRepository
  Methods: transfer(senderId: long, receiverId: long, amount: BigDecimal, key: String): TransactionResult
```

##### B. Entity Relationships
* **Aggregation:** `Wallet` has `LedgerEntry` objects (association, ledger entries can exist as records even if a wallet is soft-deleted).
* **Composition:** `IdempotencyRecord` is managed by `WalletService` transactions.
* **Association:** `Wallet` maps **One-to-Many** to `LedgerEntry`.

##### C. Applied Design Patterns
* **Strategy Pattern:** Implemented at validation layer (`LimitValidationStrategy` with concrete options like `DailyLimitStrategy` or `VIPLimitStrategy`).
* **Factory Pattern:** Used to generate matching ledger debit/credit entries inside transactional flows.

##### D. Strict Coding Rules
* Base monetary calculations utilize Java's `BigDecimal` to ensure absolute decimal precision.
* Secure token validations utilize `SecureRandom`.

#### ⚡ LLD PART 3 — EDGE CASES & SENIOR SIGNALS

1. **Transactional Outbox Pattern:** To prevent message loss if Kafka drops connections during DB updates, we write event payloads to an `outbox` table in the same DB transaction. A background poller reads the outbox and publishes messages to Kafka.
2. **Deadlock Resolution:** We sort locked wallet IDs (`LEAST(A, B)` and `GREATEST(A, B)`) before calling `FOR UPDATE` queries to prevent deadlock scenarios during concurrent mutual transfers.
3. **Ledger Reconciliation:** A nightly reconciliation worker sums up ledger entries (`SUM(DEBIT) vs SUM(CREDIT)`) for active wallets and flags balance drift alerts on mismatch.
4. **Idempotency Locks:** Idempotency checks insert a `PROCESSING` status key first. Retrying clients matching a `PROCESSING` state receive a `202 Accepted` response telling them to wait.

#### 💬 REVISION & CHALLENGE CARDS

* **Why not use FLOAT for currency?**
  > *"Floating-point systems (IEEE 754) represent decimal points by binary approximation. Adding fractions results in round-off errors (e.g., 0.1 + 0.2 = 0.30000000000000004). At scale, these minor errors compile into massive balance sheets drift. We store paise as integers or use `BigDecimal`."*

* **Why cursor pagination over OFFSET?**
  > *"OFFSET scans and discards N preceding rows before returning data. For deep pages (e.g. Page 1000), it scans 20,000 records, executing in $O(N)$ time. Cursor pagination uses index constraints (`WHERE id < :last_seen_id`), executing in $O(\log N)$ time."*

* **Why choose Pessimistic over Optimistic locking?**
  > *"Optimistic locking fails updates if a version conflict occurs, forcing a client rollback and retry. During financial transactions, balance updates are highly contentious. Making clients retry transfers on concurrent writes yields terrible user experiences. Pessimistic locking blocks requests and queues them sequentially, ensuring reliable completion."*
